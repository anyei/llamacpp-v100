- [ ] **23. SSD-streaming correctness hardening (from the 2026-07-09 code review)** — ⬜ pending, HIGH. Three verified silent-corruption hazards, none yet observed in practice but all reachable: (a) **same-pass eviction** — within one `gpu_bind`/prefill pass, a slice that was a RAM *hit* early in the loop can be evicted (`MADV_DONTNEED`) by a later insertion in the same pass when the node's expert working set exceeds `LLAMA_SSD_STREAM_BUDGET`; phase-3 H2D (or `copy_experts`) then reads zeroed pages — **silent garbage** instead of the documented "budget must cover one ubatch's union" caveat failing loudly. Fix: pin this-pass slices against eviction (or force-read every VRAM miss in phase 2); minimum: detect + abort. (b) **pread failures swallowed** — `ssd_pread_full`'s bool is ignored and the parallel workers discard returns; a slice is inserted as resident *before* its deferred read runs, so a transient disk error poisons the cache permanently. Fix: propagate failure, evict the slice from the index, abort loudly. (c) **eval-callback conflict** (`llama-context.cpp:1432`) — the sched has ONE eval-cb slot; any `cb_eval` (e.g. the `LLAMA_DEBUG` dump cb) displaces the SSD fill cb → streamed CPU-path experts never fill → silent garbage. Fix: warn at minimum; ideally chain (wrapper cb runs the fill then forwards).

(a) **No teardown**: `g_gpu_pools` (VRAM buffers), `fd_dup`, registry, and the SLRU state are process-global and never reset — server model-swap leaks VRAM+fds, and the stale SLRU can `madvise(MADV_DONTNEED)` into a *reused* arena address range of the NEXT model (cross-model corruption). - [ ] **26. Distributed: worker-local model sourcing (skip the first-load network stream)** — ⬜ pending (proposed 2026-07-09). Today the worker's share of the weights streams from the coordinator on first load (23 GB ≈ 178 s at GbE; ~10 min for the 81 GB DeepSeek); the hash cache only helps *repeat* loads. Feature: workers that have their own copy of the GGUF source their assigned tensors from local disk. **Design (no protocol change):** `rpc-server --model-dir <dir>` — the worker indexes its local GGUFs once (tensor-content hash → (file, offset, size), persisted beside `/cache`); on a `SET_TENSOR_HASH` cache miss it consults the index and reads the tensor from its local GGUF instead of requesting the stream. The coordinator keeps doing the assignment exactly as today and can't tell the difference; a mismatched/stale worker file simply hash-misses and falls back to streaming — correct by construction, no version-skew failure mode. One-time cost: hashing the local GGUF (~30 s CPU for 23 GB). Win: first load at local-NVMe speed (~6 s for an 11.5 GB slice), big models / slow networks benefit most.
