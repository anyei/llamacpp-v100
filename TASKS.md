## ⬜ Pending (approved roadmap, not started)

- [x] **9. GPU sampling in tensor-split mode** — investigated and measured, **not implemented (no headroom)**. Two findings: (a) the server disables backend sampling whenever speculation is active (`tools/server/server-context.cpp:1848`), so it can never help the MTP prod config; (b) for nospec serving the entire CPU sampling path is free — 4-conc aggregate 89.6 t/s greedy vs 89.5 t/s with full top-k/top-p/min-p chain vs 89.7 control (within noise): logits transfer + CPU sampler fully overlap GPU compute. Implementing would require a new allgather subgraph-boundary type in the meta backend (vocab-split logits hit the `handle_per_row` assert) for a measured ~0% gain.
- [x] **10. KV cache compression (TurboQuant)** — resolved: **existing quantized KV already meets the goal; TurboQuant not worth implementing.** Findings (Qwen3.6-27B, tensor mode, wikitext-2):
  - Quantized KV **works in tensor-split mode** (old note saying otherwise was stale), and this tree already ships upstream PR #21038 (commit `744c0c731`): Hadamard-rotated KV quantization — the same core trick as TurboQuant/QuaRot — auto-enabled whenever KV is quantized (opt-out `LLAMA_ATTN_ROT_DISABLE=1`).
  - PPL @32k ctx: f16 6.2904 · q8_0/q8_0 6.2922 (+0.03%, noise, **lossless, 2×**) · K q8_0/V q4_0 6.2993 (+0.14%, **2.67×**) · q4_0/q4_0 6.3093 (+0.30%, **4×**). @8k ctx all statistically identical. temp-0 128-tok gen byte-identical to f16 for q8_0 and q4_0. tg cost ≈ −3-4%.
  - Mixed K/V types crashed in tensor mode: default builds compile no FA kernel for K≠V (`fattn.cu:424`), the scheduler tries CPU fallback and the meta backend asserts (`ggml-backend-meta.cpp:557`). Fixed by building with `GGML_CUDA_FA_ALL_QUANTS=ON` (added to `.devops/cuda.Dockerfile`). Known residual: unsupported-op CPU fallback in tensor mode aborts instead of erroring cleanly.
  - TurboQuant itself (arXiv 2504.19874): no official code; needs a non-block ggml type (Lloyd-Max codebooks + per-token norms + QJL residual) and codebook gathers inside the V100 FA vec kernels; would buy ~3.5 effective bits vs q4_0+rotation's 4.5 (≈22% more capacity) — 2026 practice (SAW-INT4, vLLM) converged on rotated token-wise INT4, which is exactly what's already in-tree.
  - **Recommended prod setting: `-ctk q8_0 -ctv q8_0`** (strictly lossless, halves KV → doubles usable context/slots); `-ctk q8_0 -ctv q4_0` if more capacity is worth +0.14% ppl.

- [ ] **15. `--ssd-streaming` — three-tier weight placement (VRAM / RAM / SSD)** — ⬜ new feature, investigation + design phase. Run models whose weights don't fit in VRAM (or even VRAM+RAM) by streaming weight tensors from SSD on demand, as a third managed placement tier that composes with the existing two.
  - **UX / CLI semantics (to finalize in design):** three combinable tiers — resident-GPU (`-ngl N`), resident-CPU (today's host layers), and streamed-SSD (new). `--ssd-streaming` alone = stream everything (min-memory mode); `-ngl N --ssd-streaming` = N hot layers resident in VRAM, rest streamed; all three = resident GPU slab + resident CPU slab + streamed tail. Likely knobs: `--ssd-stream-budget <MiB>` (pinned staging-ring size), placement policy (which layers get which tier — front-loaded resident vs access-frequency), and a per-tensor override story consistent with `-ot`/`--override-tensor`.
  - **Physics / honest expectations (measured on this box 2026-07-07):** the models NVMe sustains **1.6 GB/s** O_DIRECT sequential reads (`dd iflag=direct`, 4 GiB from a resident GGUF). A dense 27B Q4 (16.4 GB) fully streamed touches all bytes every token → **~0.1 t/s decode ceiling**; this is a *runnability* feature, not a speed feature, for dense decode. The economically interesting cases: (a) **prefill/batch amortization** — one layer stream serves the whole ubatch, so a 4096-token prefill pays the 16 GB stream once (~10 s) across thousands of tokens; (b) **MoE expert streaming** — only route-active experts per token, plus a hot-expert LRU cache in RAM/VRAM (GLM-class A3B: ~3B active of ~30B total → ~10x fewer bytes/token than dense); (c) **mostly-resident mixes** — e.g. 2 streamed layers out of 48 adds only ~0.4 GB/token ≈ +0.25 s/token.
  - **Investigate first (phase 0):** what the feature must beat is *unmanaged* SSD access, which llama.cpp already has: default mmap demand-pages CPU-side weights from disk, so an oversized model already "runs" via page-cache thrash. Baseline-measure that (force with a cgroup memory limit) so managed streaming has a number to beat. Survey: `-ot` buffer-type overrides (partial prior art for placement), `--no-mmap`/direct-IO paths, upstream issues/PRs on disk offload, FlexGen & DeepSpeed-Inference literature (their pipelining/placement math applies directly), GPUDirect Storage (cuFile) driver/filesystem requirements on V100. Deliverable: `docs/ssd-streaming-plan.md` with the design + measured baselines (mirror the distributed-inference-plan.md pattern).
  - **Design sketch (to validate in phase 0):** a *streamed* backend buffer type whose tensors are backed by `(file, offset)` instead of resident memory; a pinned host staging ring sized by `--ssd-stream-budget`; prefetch driven by graph execution order — double-buffer layer n+1's tensors while layer n computes (the graph-plan/scheduler hook point is the same place the decode-graph cache lives, so per-layer prefetch lists can be precomputed per cached graph shape); reads via O_DIRECT (+ io_uring) so streaming does not evict the resident CPU tier from page cache; no eviction complexity (weights are immutable, read-only); dequant stays on GPU exactly as today. GGUF is already offset-addressable per tensor — no file-format work needed.
  - **Phased plan:** 0 = investigation + baselines + plan doc (above); 1 = CPU-tier managed streaming (prefetch thread, staging ring), gate: byte-identical output + beats mmap-thrash tg measurably; 2 = GPU-tier streaming (staging ring → async H2D on a side stream overlapped with layer compute); 3 = placement policy + CLI + docs + compose profile; 4 = MoE-aware expert streaming with hot-expert cache; 5 (stretch) = cuFile/GDS direct SSD→VRAM, and interplay with distributed (task 12): worker-side streaming would let a TP island serve models bigger than the island's combined VRAM.
  - **Verification (house rules):** temp-0 byte-exactness vs fully-resident runs at every phase; enforced ceilings (RSS + VRAM measured under load); a tg/pp table per tier mix in docs/perf-tuning-v100.md.
