- [x] **9. GPU sampling in tensor-split mode** — investigated and measured, **not implemented (no headroom)**. Two findings: (a) the server disables backend sampling whenever speculation is active (`tools/server/server-context.cpp:1848`), so it can never help the MTP prod config; (b) for nospec serving the entire CPU sampling path is free — 4-conc aggregate 89.6 t/s greedy vs 89.5 t/s with full top-k/top-p/min-p chain vs 89.7 control (within noise): logits transfer + CPU sampler fully overlap GPU compute. - [x] **10. KV cache compression (TurboQuant)** — resolved: **existing quantized KV already meets the goal; TurboQuant not worth implementing.** Findings (Qwen3.6-27B, tensor mode, wikitext-2):
  - Quantized KV **works in tensor-split mode** (old note saying otherwise was stale), and this tree already ships upstream PR #21038 (commit `744c0c731`): Hadamard-rotated KV quantization — the same core trick as TurboQuant/QuaRot — auto-enabled whenever KV is quantized (opt-out `LLAMA_ATTN_ROT_DISABLE=1`).

  - **Recommended prod setting: `-ctk q8_0 -ctv q8_0`** (strictly lossless, halves KV → doubles usable context/slots); `-ctk q8_0 -ctv q4_0` if more capacity is worth +0.14% ppl.

The economically interesting cases: (a) **prefill/batch amortization** — one layer stream serves the whole ubatch, so a 4096-token prefill pays the 16 GB stream once (~10 s) across thousands of tokens; (b) **MoE expert streaming** — only route-active experts per token, plus a hot-expert LRU cache in RAM/VRAM (GLM-class A3B: ~3B active of ~30B total → ~10x fewer bytes/token than dense); (c) **mostly-resident mixes** — e.g. 2 streamed layers out of 48 adds only ~0.4 GB/token ≈ +0.25 s/token.
